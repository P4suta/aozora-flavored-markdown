# 0017. Derive the TypeScript `.d.ts` with `tsify`

- Status: accepted
- Date: 2026-06-21
- Deciders: @P4suta
- Tags: ir, typescript, codegen, wasm, dx

## Context

The IR + wasm-envelope TypeScript surface (consumed by the playground and
aozora-flavored-markdown-obsidian) used to be generated by a bespoke xtask
command (`crates/xtask/src/types.rs`): ~520 lines of hand-written `.d.ts`
string assembly, kept "honest" against the Rust types by three custom
mechanisms — `assert_*_variants` compile-time witnesses, a field-completeness
test, and a tag test — plus a byte-compare drift gate (`just types-check`) over
a committed `aozora_flavored_markdown_types.d.ts` artefact.

That machinery was substantial and only self-consistent: the drift gate checked
the committed artefact against the *hand-written* strings, not against the
actual Rust shapes, so a renamed field that the author forgot to mirror could
still slip through between the witnesses. The IR types already derive
`serde::Serialize` with the camelCase / `kind`-tagged attributes the TS needs,
and the project prefers mainstream tooling over bespoke machinery. For a
wasm-bindgen crate the idiomatic, modern choice is `tsify`, which derives the
TypeScript straight from the Rust types **and** wires it into the wasm ABI so
the boundary itself is typed.

## Decision

Generate the `.d.ts` with [`tsify`](https://crates.io/crates/tsify)
instead of hand-written codegen. (`tsify` is the maintained crate; the
`tsify-next` fork that briefly carried development has merged back upstream
and is now deprecated — RUSTSEC-2025-0048.)

- The IR + diagnostic types (`aozora_flavored_markdown::ir`,
  `aozora_flavored_markdown::diagnostics`) derive `tsify::Tsify` behind a
  feature-gated `#[cfg_attr(feature = "tsify", …)]`. The lib's `tsify` feature
  (off by default) pulls `tsify` + `wasm-bindgen`; only
  `aozora-flavored-markdown-wasm` enables it, so host builds (CLI / epub /
  tests) never pull wasm-bindgen.
- The wasm wire-envelope structs (`RenderResult`, `RenderOptions`,
  `BlockResult`, `BlocksResult`) derive `Tsify` with `into_wasm_abi` /
  `from_wasm_abi`, so the `#[wasm_bindgen]` entry points are typed end to end:
  `render` returns `RenderResult` and accepts `Option<RenderOptions>` directly,
  with no `serde_wasm_bindgen` hand-conversion and no `as unknown as …` cast on
  the JS side.
- `wasm-pack build` emits the `.d.ts` next to the `.wasm`; the playground and
  obsidian import the types from the generated package.

Removed: `crates/xtask/src/types.rs` and the `Types` xtask subcommand; the
committed `.d.ts` artefact; the `just types` / `just types-check` recipes and
the CI `types-check` job; and the `assert_*_variants` witnesses in
`ir::types` (their sole job was nudging the hand-written union, which no longer
exists). The camelCase wire-string test (`classification_enums_serialize_to_stable_wire_strings`)
stays — it locks the serde output `tsify` now reads.

Two shape fixes fell out of the move:

- Recursive IR fields used `Vec<Self>`; `tsify` emits the literal `Self[]`
  (invalid TS), so they are now spelled `Vec<IrBlock>` / `Vec<IrInline>`
  (semantically identical in Rust).
- `BlockResult.source_line` had no `rename_all`, so it serialised as
  `source_line` while the docs/old `.d.ts` promised `sourceLine`. It now carries
  `#[serde(rename_all = "camelCase")]`, aligning the runtime wire format with
  the documented `sourceLine` contract.

## Consequences

- **Single source of truth.** The `.d.ts` is derived from the Rust types, so the
  TS can no longer drift; ~700 lines of bespoke generator + tests + gate are
  deleted.
- **Typed wasm boundary.** Mistyped IR access in the playground / obsidian is a
  compile error, not a runtime surprise.
- **A new feature-gated coupling.** The lib gains optional `tsify` +
  `wasm-bindgen` deps. They are off by default and behind the `tsify` feature,
  so the host dependency graph is unchanged; `cargo-shear` sees them as used
  (via the wasm crate + the optional declaration).
- **Downstream recompiles.** aozora-flavored-markdown-obsidian must regenerate
  its types from the new wasm package — already the expected cost of a wasm-crate
  version bump (see the crate's stability note). The wasm export `renderAfm`
  was also renamed to `render` in the same pass (post-rebrand cleanup, ADR-0016).

## Alternatives considered

**`ts-rs`.** Derives `.ts` from the types but does not integrate with
wasm-bindgen, so the wasm boundary would stay `JsValue` + casts and need its own
export/diff gate. Lighter coupling (no wasm-bindgen in the lib) but it leaves the
boundary untyped — the bigger win here.

**Keep the hand-written generator.** Rejected: bespoke, ~700 lines, and only
self-consistent (see Context). Exactly the "don't invent machinery mainstream
tooling already provides" case.

**`typeshare`.** A CLI-driven, multi-language generator. Overkill for a
single-target (TypeScript) wasm crate and adds an external tool to the image.
